# AI-assisted Changelog

One line per significant AI-assisted improvement (newest first).

- 2026-07-19: **Primitive library for scripted models вЂ” a scripted model now beats the hardcoded one.** `cubic_root(a,b,c,d)` and `quadratic_root(a,b,c)` are registered as script functions in BOTH engines (ExprTk via `ifunction`, ChaiScript via `chaiscript::fun`), so a script gets an equilibrium concentration without hand-rolling an iteration. New preset "Binding 1:1/1:2 вЂ” cubic_root primitive" is loop-free. Measured (release, min of 5): hand-written Newton in ExprTk 803.4 ns -> cubic_root() 332.3 ns per point-eval = **2.4x**, and 0.78x of the hardcoded `nmr_ItoI_ItoII` (427.1 ns with its default Newton cubic; 255 ns with the analytic one вЂ” so at equal primitives scripting costs ~30%, not the 2-8x measured without a library). Two bugs found by the new tests while building this: `AnalyticCubicRoot` is badly conditioned for binding-equilibrium coefficient ranges (a ~ 1e11 against d ~ -1e-4) and now Newton-polishes its closed-form estimate; and the first preset passed equil.h's coefficients, which use the STEPWISE K12, while the presets carry CUMULATIVE betas вЂ” the cubic is `b12 G^3 + (b11 + b12(2H0-G0)) G^2 + (1 + b11(H0-G0)) G - G0`. `test_cubicsolver`'s constant grid was widened to lgK12 = 9 (the failure only appears above 1e5) and its residual check is now relative to the term magnitudes, skipping the degenerate G0 = 0 point where the root is exactly 0.
- 2026-07-19: **Scripted 1:1/1:2 solver 2.7x faster** (2240 -> 839 ns per point-eval). The preset ran a fixed 60-step bisection, which converges linearly (one bit per step). Replaced by a SAFEGUARDED Newton written in ExprTk вЂ” Newton step when it stays inside the bracket, bisection otherwise вЂ” so it keeps the unconditional convergence but needs ~5 iterations. A well-written scripted solver is now 1.93x the hardcoded model per evaluation (was 3.5x) and 1.12x when it calls the native solver.
- 2026-07-19: **ChaiScript ported to the ScriptingEngine interface and evaluated.** New `ChaiEngine` (`chaiengine.h`, flag `_Models`) parses the equation to an AST ONCE and re-evaluates it, with every bindable name added as a ChaiScript global referencing a stable C++ double вЂ” the fair counterpart to ExprTk's slot binding, and the opposite of the legacy ChaiInterpreter which re-eval'd a string-substituted expression per data point. Selectable via the model `Engine` field; `test_scriptmodel` asserts both engines agree to 1e-12 on the same equation. Measured (release, min of runs, same data): small expression 241 ns ExprTk vs 440 ns ChaiScript (1.83x); large expression 264 ns vs 2234 ns (8.45x). ExprTk is nearly insensitive to expression size (+10%) while ChaiScript grows 5x вЂ” tree-walking interpreter vs compiled expression. ExprTk stays the default. Note the vendored ChaiScript math extras define bootstrap() out of line, so it must be included by exactly one TU; `SupraFitChaiMathModule()` in chaiinterpreter.cpp is now that single point (a second include caused a multiple-definition link error).
- 2026-07-19: **Closed-form cubic root solver as an opt-in variant** (`CubicSolver::setMethod`, `AnalyticCubicRoot`). `MinCubicRoot` is not a closed form despite the name вЂ” it runs three Newton searches to pick a root вЂ” and it has two real defects: the third "root" iterates on the DERIVATIVE using the derivative as its own derivative, so the Newton step degenerates to `x -= 1` (a linear scan, capped at 100 steps) and the result is not a root at all; and two branches of its root selection are unreachable, so it returns the first non-negative of roots 1/2 rather than the minimum. Added a Cardano/trigonometric implementation returning the smallest non-negative root. It is selectable and **defaults to the historical Newton path**, so published fit results stay reproducible. Measured on `nmr_ItoI_ItoII` (min of 5 release runs): 439.9 -> 255.3 ns per point-eval = 1.72x; together with the FillDesign fix 620.7 -> 255.3 ns = 2.43x. It is also more accurate: at G0 = 0 the analytic solver returns 0 where Newton returns ~3e-9, because Newton stops at |f(x)| < 1e-8 ABSOLUTE regardless of scale. New ctest `test_cubicsolver` (residual check on both discriminant branches, agreement with Newton within its own 1e-8 accuracy over a grid of titration cubics, default-is-Newton guard). `benchmark_scriptengine analytic` switches the variant on.
- 2026-07-19: **`nmr_ItoI_ItoII` concentrations 1.44x faster** (620.7 -> 431.0 ns per point-eval, min of 5 release runs, A/B against the unmodified file). `FillDesign()` solved TWO cubics per data point вЂ” `HostConcentration` and `GuestConcentration` вЂ” and built a `QList<qreal>` of the constants twice per point inside the fit hot loop. Note `MinCubicRoot` is not a closed form despite the name: it runs three separate Newton searches (one per root) to pick the smallest, so the model was doing six root searches plus two heap allocations per point вЂ” which is why the general numerical `nmr_any` outran it. Now the constants list is hoisted and only the guest cubic is solved; the free host follows from its own mass balance H0 = [H](1 + K11[G] + K11K12[G]^2), the very relation the guest cubic is derived from вЂ” exact, and mass-balance consistent by construction instead of two independently converged Newton solves. `test_nmr_any_equivalence` (which validates this model against nmr_any) and `test_varpro` unchanged. Remaining lead: `MinCubicRoot` computes all three roots when only the smallest positive one is used, and it is shared by the other fixed titration models.
- 2026-07-19: **`BFGSConcentrationSolver` renamed to `ConcentrationSolver`** (`bfgsconcentrationsolver.{h,cpp}` -> `concentrationsolver.{h,cpp}`, `test_bfgs_solver` -> `test_concentrationsolver`, ctest `ConcentrationSolverTest`). The name was misleading: the default method has been a damped Newton iteration with the analytic Hessian (`Method::LevenbergMarquardt`) since 2026-07-11; BFGS survives only as the legacy option, and `Method::BFGS` / the "bfgs" config string are unchanged. Also: `benchmark_scriptengine` now measures VarPro alongside LevMar, which corrects the earlier conclusion вЂ” VarPro speeds up `nmr_any` ~4x (542->137 ms MC) and makes it the fastest configuration, so a scripted model sits ~5x behind the BEST hardcoded setup rather than the ~9% measured against a LevMar-only baseline. The gap is VarPro, not the scripting: scripted models cannot declare that they are linear in their local parameters, so they never get the projection (the Sum-c-epsilon and NMR-shift presets ARE linear in their locals вЂ” an obvious next feature). Fixed a pre-existing break found on the way: `test_pipeline` did not link because `CLIENT_SOURCES` lost `data_factory.cpp` when DataFactory was split out of SupraFitCli.
- 2026-07-19: **Native equilibrium solver callable from a script + engine benchmark.** `ScriptingEngine::setSpeciation()/setSpeciationPoint()` hand a `SpeciationEngine` to the backend; `ExprTkEngine` registers `spec_solve(t0,t1,вЂ¦)` (variadic `igeneric_function`), `spec_free(i)` and `spec_conc(j)`, so an equation can use the native speciation solver instead of iterating in the interpreter (that solver defaults to a damped Newton method with the analytic Hessian вЂ” `Method::LevenbergMarquardt`; the class is only called `BFGSConcentrationSolver` for historical reasons and BFGS is the legacy option). `ScriptModel` builds it from an optional `Reactions` field and pushes the first N globals as lg beta; the per-point index is forwarded so the solver warm-starts (worth 5.3x on its own вЂ” without it the native path was SLOWER than the scripted bisection). New preset "Binding 1:1/1:2 вЂ” native solver" mirrors the scripted-bisection preset exactly, verified equal in `testNativeSolverPreset`. New manual perf tool `benchmark_scriptengine` (release build!) measures the 1:1/1:2 system against the hardcoded `nmr_ItoI_ItoII`: per residual evaluation 1.00x hardcoded / 0.40x nmr_any / 3.48x scripted-bisection / 0.82x scripted-native; Monte Carlo (100 steps, full fits via Clone()) 1.00x / 0.80x / **16.6x** / **1.09x**. Conclusion: with the numerics kept native, a scripted model costs ~9% against hardcoded C++, while solving the mass balance inside the interpreter costs 16x.
